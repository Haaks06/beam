What Beam actually is

A 2-device, 2-minute, no-account way to beam a photo or link from one device to another. No cloud storage, no history, no sign-up. This page is a working reference, not marketing — every claim on it is either the real running code or a button that runs the real code live.

The mental model

🔗 Exactly two devices

A "pairing" is never more than two devices. One creates it, the other joins with a code or QR scan. There's no concept of a third device, a group, or a saved contact.

⏱️ Exactly two minutes

The instant the second device joins, a 2-minute clock starts. A background sweep (every 5 seconds) destroys everything about the session the moment it expires — no grace period.

🚫 No accounts, no history

There used to be full username/password accounts, friend lists, and an admin panel. All of it was deliberately removed — see Patch Notes — in favor of this simpler, disposable model.

↔️ Bidirectional once paired

Either device can Send or Receive at any point during the session — nobody is locked into one role for the whole 2 minutes.

The high-level flow

  1. Someone taps "Send" or "Receive"

    Send generates a fresh inbox + a pairing code immediately (POST /inbox then POST /pair/init). Receive just opens a form to enter or scan a code — it doesn't create anything until a code is actually claimed.

  2. The other device joins

    Scanning the QR (or typing the code) hits POST /pair/claim. The instant a second device is present, the inbox is stamped with paired_at and a fixed expires_at exactly 2 minutes later.

  3. Either side sends something

    A link/text or a photo — see the dedicated "How a File Moves" page for the full breakdown of exactly what happens here.

  4. The window closes

    2 minutes after pairing, everything about the session — items, uploaded files, both device tokens, the inbox itself — is permanently deleted. Both devices get pushed a session-expired event over their live connection so the UI reacts instantly instead of just going quiet.

How a file actually moves

Every stage a beamed photo passes through, from the moment you pick it to the moment it's permanently gone. Each step names the real file, table, or function responsible — not a simplified stand-in.

  1. 1. You pick or drop a photo

    The Send tab's drop zone accepts a click-to-choose, a drag-and-drop, or a clipboard paste. The file never leaves the browser until you press "Send photo" — picking it only shows a local preview via URL.createObjectURL.

    web-client/src/app.js — setSelectedPhoto(), photoDropZone drag/paste handlers

  2. 2. Upload — POST /items/photo

    A multipart request, authenticated by the device's bearer token (requireToken middleware). This is the moment the file actually leaves your device.

    relay-server/routes/items.js — router.post('/photo', upload.single('file'), ...)

  3. 3. Validation, before a single byte is trusted

    Only image/jpeg, png, webp, or heic are accepted — anything else is rejected outright. A hard 15 MB cap applies. Crucially, the client's original filename is discarded entirely — the server generates its own random name.

    relay-server/lib/upload.js — ALLOWED_MIME_EXT, MAX_UPLOAD_BYTES=15MB, filename: crypto.randomUUID() + extension (this is what closes a path-traversal / overwrite vulnerability — the client never controls the on-disk name)

  4. 4. The file is written to disk

    Saved under UPLOAD_DIR — in production, a persistent Fly.io volume mounted at /data. This is a genuinely new file on disk, not a database blob.

    Example real path: /data/uploads/9f40a2da-a76d-452d-8580-6cde23d79705.jpg

  5. 5. A row is written to the items table

    The row records inbox_id (who can see it), from_device_id (which device sent it — used to make sure the sender never sees their own item echoed back), the generated file_path, mime type, and a timestamp. No original filename, no metadata beyond what's needed to serve it back.

    relay-server/db.js — items(id, inbox_id, type, content, file_path, mime_type, source_label, created_at, from_device_id)

  6. 6. Live delivery — if the other device is connected right now

    The relay pushes the new item over Server-Sent Events to every open connection on that inbox except the device that just sent it — so you don't see your own upload echo back into your own "Received" list.

    relay-server/lib/sse.js — broadcast(inboxId, item, excludeDeviceId)

  7. 7. Catch-up delivery — if it wasn't connected

    SSE has no delivery guarantee across a dropped connection or a closed tab. When the receiving device (re)connects, it calls GET /items?since=<lastSeenId>, which returns anything it missed — again, excluding whatever this same device sent, for the same reason as live delivery.

    relay-server/routes/items.js — WHERE inbox_id = ? AND id > ? AND (from_device_id IS NULL OR from_device_id != ?)

  8. 8. Actually viewing/downloading it

    The photo itself isn't embedded in the item's JSON — the client fetches GET /items/:id/file separately, with its bearer token. The server checks the item's inbox_id matches the token's inbox before streaming a single byte — a stolen item id alone gets you nothing.

    relay-server/routes/items.js — getItem.get(req.params.id, req.device.inbox_id) — 404 if it doesn't match

  9. 9. The 2-minute clock runs out

    A background sweep runs every 5 seconds, checking for any inbox whose expires_at has passed. When it finds one, it deletes — in this order — every item row, unlinks every uploaded file from disk, deletes both device rows (revoking both tokens instantly), then deletes the inbox row itself.

    relay-server/lib/sessionCleanup.js — sweep() → destroyInbox(inboxId)

  10. 10. Gone. Genuinely gone.

    Not soft-deleted, not archived — the row and the file are both physically removed. Both devices immediately get a session-expired push (not just a timeout on their own countdown) so the UI reacts the instant the server-side deletion happens, not whenever the client's own clock catches up.

Security posture

      
What's actually enforced today, stated plainly — including the things this deliberately does not defend against, because the ephemeral model makes them moot.


      
What's enforced

      

        

          
🔑 Bearer token auth enforced

          
Every route except account creation/claiming requires a valid device token, checked against the devices table on every request (requireToken middleware).

        

        

          
📦 Inbox-scoped isolation enforced

          
Every query is scoped by inbox_id, not just a row id. A guessed or stolen item id from a different pairing returns 404, not someone else's data.

        

        

          
🗂️ Path-safe uploads enforced

          
The client's filename is never trusted or used — the server always generates its own (crypto.randomUUID()), closing path traversal and overwrite attacks outright.

        

        

          
🖼️ MIME allowlist + size cap enforced

          
Only jpeg/png/webp/heic accepted; 15 MB hard limit per file (relay-server/lib/upload.js).

        

        

          
🔗 XSS-safe link rendering enforced

          
A sent "link" only ever renders as a clickable <a href> if it actually parses as http: or https:. Anything else (including something like javascript:alert(1)) renders as inert plain text, never as a link target.

        

        

          
🚦 Rate limiting, with readable errors enforced

          
/inbox: 120/hour · /pair/*: 20/min · /items/*: 120/min, all per-IP. All three return real JSON on a 429, not the plain-text default that used to crash the client with a parse error.

        

        

          
⏳ Guaranteed data expiry enforced

          
Nothing survives past 2 minutes after pairing (or 10 minutes if a code is generated and never claimed) — enforced by a background sweep, not by trusting clients to clean up after themselves.

        

        

          
🌐 CORS + Helmet headers enforced

          
Standard security headers via helmet; CORS origin restricted via CORS_ORIGIN env var when set, permissive only in local dev.

        

      


      
What this deliberately doesn't have (and why that's fine here)

      

        
Not presentWhy it's an acceptable tradeoff

        
Accounts / passwordsRemoved on purpose — see Patch Notes. No credentials means no credential-stuffing, no password database to leak, nothing to phish for.

        
Long-term historyThere's nothing to breach two minutes after any given pairing — the data simply doesn't exist anymore.

        
End-to-end encryptionTraffic is encrypted in transit (HTTPS), but the relay itself can technically see file contents for the ~2-minute window they exist. Worth knowing if that matters for your use case.

        
Abuse-proof rate limitingPer-IP limiting is imperfect behind CGNAT/shared IPs — accepted tradeoff for keeping the limits generous enough not to block real usage (this has already been raised twice this session after real false positives).

      


      
A real incident, for transparency

      
Deploying the ephemeral-pairing rewrite briefly took production down: node:sqlite enforces foreign keys by default, and a leftover table from the old accounts system still referenced deleted inbox rows, crashing the cleanup sweep every 5 seconds until Fly.io's max-restart limit was hit. Found, fixed, and covered with a regression test reproducing the exact crash within the same session — see Patch Notes for the full writeup. Documented here rather than glossed over.

    


    
    

      
Functions & API reference

      
Every route the relay exposes, what it needs, and what it does.


      
Pairing

      

        
RouteAuthPurpose

        
POST /inboxNoneCreates a new inbox and mints the first device's token in one call. This is the ONLY place a new inbox is ever created.

        
POST /pair/initBearer tokenGenerates a pairing code + QR for a second device to join this inbox. Refuses if the inbox already has 2 devices.

        
POST /pair/claimNone (can't have a token yet)Exchanges a valid, unused, unexpired code for a device token. The second device joining starts the 2-minute clock.

        
GET /pair/status/:codeNoneLets the initiating device poll whether its code has been claimed yet, without exposing the resulting token.

      


      
Sending & receiving

      

        
RouteAuthPurpose

        
POST /items/linkBearer tokenSends a link or arbitrary text. Broadcasts live to the other device, excluding the sender.

        
POST /items/photoBearer tokenMultipart photo upload — see the File Flow page for the full pipeline.

        
GET /items?since=idBearer tokenBacklog/catch-up fetch for anything missed since a given item id — used on reconnect.

        
GET /items/:id/fileBearer tokenStreams a photo's actual bytes. Scoped to the caller's own inbox.

        
GET /eventsBearer token (query param, since EventSource can't send headers)The live Server-Sent Events stream — pushes new items and a session-expired signal.

      


      
Misc

      

        
RouteAuthPurpose

        
GET /healthNoneReturns { ok, version } — how every deploy in this project gets confirmed live.

      


      
Client-side building blocks

      

        
web-client/src/app.js
The entire UI's logic — session state machine (start → invite → active → ended), the photo drop zone, the in-browser QR camera scanner (jsQR), SSE handling.

        
relay-server/lib/sse.js
The pub/sub layer — tracks subscribers per inbox, supports excluding a specific device id from a broadcast.

        
relay-server/lib/sessionCleanup.js
The 5-second sweep that makes "ephemeral" actually true, not just a claim in the UI copy.

        
relay-server/lib/upload.js
Multer configuration — MIME allowlist, size cap, safe filename generation.

        
desktop-app/main.js
Electron shell — just a BrowserWindow pointed at the live relay URL, plus a tray icon and native save/notify bridge for received items.

        
desktop-app/preload.js
The bridge exposing window.beamNative — also how the shared web-client code detects "I'm running inside the desktop app" (used to hide the camera scanner there, since Electron's camera permissions aren't wired up).

Live-usage fixes

        
    
          
  • Fixed (properly, third attempt): the inbox-creation rate limit kept getting hit during normal use even after two straight cap increases (5/hr → 30/hr → 120/hr). The real problem was the 1-hour window — hitting the cap meant up to an hour locked out. Switched to a 10-minute window with a 200 cap: harder to hit, and recovers in minutes if it happens anyway.
    • 
          
  • Fixed: refreshing the page mid-session (e.g. on a phone) could wipe out a perfectly valid active connection — a leftover URL parameter was re-triggering a pairing claim on every reload.
    • 
          
  • Fixed: rate-limit errors crashed the client with a raw JSON-parse error instead of a readable message.
    • 
          
  • Fixed: the sender used to see their own sent item echoed back into their own Received tab. Now excluded server-side; the send confirmation names exactly what was sent instead.
    •
